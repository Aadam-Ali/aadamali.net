Site Design

2023-07-02 - 4 minutes - 684 words

As promised in my first post , here is the post about the architecture (and design) of this site. I’m going to talk about the tools and design choices made that made this webiste possible.

Hugo

My first decision was to use a static site generator (SSG) - this was to avoid the overhead of having to manage the HTML and CSS for each page, allowing me to concentrate on actually writing the content. This is also helped by the fact that in most SSGs posts are written in Markdown rather than HTML, which makes writing content much easier.

Choosing an SSG was fairly straightforward, I wanted to use something that was well documented and had a large community. This left me with two choices, Hugo and Jekyll . I ended up choosing Hugo for two main reasons, I liked the fact that it was a single binary and didn’t require me to install anything extra on my system for example Jekyll requires Ruby to be installed on your system. The second reason was that Hugo seemed to be the faster of the two options.

From my experience so far Hugo is pretty easy to use. I was able to set up this website’s components (header, footer, title sections, etc.) and layouts - single pages (like the one you’re reading) and lists, e.g. /blog - within a couple of hours.

Two Repos

This website is managed using two repositories, blog and aadamali.net .

The blog repo contains the Hugo directory structure and a Makefile (see below):

archetypes/  content/  hugo.toml  Makefile  resources/  themes/
assets/      data/     layouts/   public/   static

This is a private repository that I do not intend on making publicly available. However, if there is interest in the theme of this site, I am happy to create a separate repository containing the Hugo theme.

aadamali.net is a public repository that contains the static content that you are currently viewing. This is a submodule of the blog repository under the public/ directory. To build the content I run the make command from the blog repository, I then commit and push the static content in public/ to GitHub - at some point I’ll automate this using GitHub Actions.

GitHub Pages

In terms of hosting I wanted to start with something cheap or free for two main reasons. Firstly, the content on the site is static and does not require any complex infrastructure. Secondly, I did not want to invest into something that I had not fully developed the habit of contributing to.

What better way to do this than to use GitHub Pages, which I enabled on my aadamali.net repository and configured to automatically create a new deployment on pushes to the main branch.

Design

The design is something that I wanted to be personal yet simple. That is why I didn’t want to use one of the many community created Hugo themes that are freely available but rather write my own CSS.

From my point of view there are two main components to this design, the colour scheme and the font.

In terms of colours I decided to use the Gruvbox Dark mode palette because I use it on my machines, mainly as my terminal colour scheme, as it is easy on the eyes. However, this is very much subject to change as I didn’t want to spend too much time on design, so I went with a predefined colour palette.

JetBrains Mono is the font I have chosen and similar to the colour scheme I use it in the terminal as well as text editors / IDEs. I’m conflicted in this area as I don’t think the monospace look is to everyone’s taste and am considering switching to something like Fira Sans or Fira Sans Condensed as they’re in the same family as Fira Code and Fira Mono , two fonts that I use alternatively to JetBrains Mono.

Closing

Thank you for taking the time to read this post. Please feel free to contact me with your thoughts on the font choice, or with topics you’d like me to write about in the future.
